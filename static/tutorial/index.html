01导览

项目概览

本项目的核心思路：不再依赖官方 API，而是自动化你本地的浏览器——让脚本在你已登录的 AI 网站上输入问题、等待回复、提取结果，再按 OpenAI 的数据格式返回给任何客户端。

OpenAI / Anthropic 双协议兼容 多站点 · 多标签页并发 流式输出 图片 / 音频 / 视频回传 自动化恢复命令 函数调用转译

典型用途：统一本地接入方式；按站点或标签页组织不同工作流；在长文本场景下通过文件粘贴承载上下文；在调试时观察页面侧与请求侧的执行流程。

先搞清一件事：脚本运行后会有两个浏览器

受控浏览器由脚本自动启动，用来登录 AI 网站并真正收发消息；你自己的普通浏览器只负责看文档、日常上网。脚本运行时请不要手动操作受控浏览器——窗口可以最小化，但不要缩小边框、折叠页面元素或切换标签页。

02导览 · 必读

来自作者：使用预期

这七条来自作者的维护说明，能帮你在遇到问题时少走 90% 的弯路。

1 · 站点突然不能用了？优先检查选择器

网站改版后最易失效的是选择器（告诉脚本输入框、发送按钮、结果容器在哪的 CSS 规则）。大多数故障检查选择器即可，而非解析器。标准修法：

  1. 在受控浏览器里对目标元素右键 →「检查」。
  2. 把元素树截图或复制出来发给 AI，让它写出新的选择器。
  3. 回控制台粘贴替换，重新测试。

如果不是选择器问题，才轮到提取器 / 解析器——那需要拦截网络请求、分析响应结构，难度高得多。

2 · 各站点的维护覆盖范围

Gemini、DeepSeek 是作者日常使用最多、维护最及时的站点。其它站点不保证第一时间跟进，遇到问题更推荐加 QQ 群反馈，通常比 Issue 更快。

3 · Function Calling 尚不稳定

它比较依赖模型本身的理解能力，可能存在一定问题。详见「函数调用」章节的预期管理。

4 · 遇到问题，优先让 AI 帮你分析

选择器失效、配置写错、工作流顺序不对——把现象、日志截图、元素树交给 AI，通常比等人工回复快得多。

5 · 反馈 Bug 的正确姿势

说清做了什么、期望什么、实际看到什么，附上 DEBUG 级别日志。表述不清时，先让 AI 帮你整理一版再发。

6 · 修好了欢迎分享回来

不管是更新的选择器还是补好的配置文件，分享给作者或群友都能让整个社区少走弯路。

7 · 项目的核心场景

本项目围绕酒馆 / RP（角色扮演）场景设计打磨。其它场景能用，但维护上偏向"能用就用，不保证持续深耕"。

03上手指南

快速开始

第一次使用，按下面四步走即可跑通整条链路。

  1. 启动脚本。双击运行 start.bat，脚本会自动打开受控浏览器，默认使用项目内的 chrome_profile/ 目录；第一次启动通常是空白、未登录状态。
  2. 登录目标站点。在受控浏览器中打开目标 AI 网站并完成登录，直到进入真正可以对话的页面。以 Gemini 为例，建议打开 https://gemini.google.com/，并保持左侧侧边栏处于展开状态。
  3. 清理无关标签页。正式发请求前，确认受控浏览器里只保留目标 AI 网站——搜索页、邮箱、教程页等无关标签页都可能干扰脚本自动操作。
  4. 连接客户端。按「连接 API」章节的参数填写地址、密钥与模型名；随后打开控制面板 http://127.0.0.1:8199/ 即可查看日志与配置。

04上手指南

连接 API

本项目提供兼容 OpenAI 格式 的接口，绝大多数支持"自定义 OpenAI 端点"的客户端都能直接使用。

服务提供商 Provider

选择 OpenAI / OpenAI 兼容 / 自定义。不同客户端叫法不同，选那个能填写自定义地址的即可。

API 密钥

未启用认证（AUTH_ENABLED=false）时填占位值如 sk-local 即可。启用后必须与 .env 的 AUTH_TOKEN 一致；控制面板密钥 DASHBOARD_AUTH_TOKEN 是另一套，不要提供给 API 使用者。

Base URL 形态地址适用场景
默认 · 自动分配http://127.0.0.1:8199/v1不关心具体站点或标签页时使用
指定站点域名http://127.0.0.1:8199/url/gemini.com/v1自动匹配该站点的标签页
指定标签页http://127.0.0.1:8199/tab/1/v1编号见「标签页池」，适合绑定过预设的标签页
指定完整 URLhttp://127.0.0.1:8199/tab-url/{token}/v1token 从「标签页池」复制，严格匹配当前 URL
站点 + 预设路径http://127.0.0.1:8199/url/gemini.com/pro/v1同一站点拆多种用途时最推荐，直接作为 Base URL
完整路径补全在末尾补上 /chat/completions部分客户端要求填写完整接口路径

模型名称 (Model)：建议填写便于客户端保存配置的占位名（如 web-api、gemini-web）。实际响应来源由当前打开的站点与预设决定。

测试步骤

把地址、密钥、模型名填好后，直接发一句正常对话进行测试。不建议点客户端里的"测试 API"按钮——酒馆（SillyTavern）的测试按钮发送的请求过短，部分站点不会正常响应。

单次请求指定预设的三种方式

聊天接口支持直接指定 preset_name，只对当前这一次请求生效，不改变站点默认预设。

bash
# 方式 A：直接写进路径（最推荐，适合作为 Base URL）
curl "http://127.0.0.1:8199/url/gemini.com/pro/v1/chat/completions" ^
  -H "Content-Type: application/json" ^
  -d "{\"model\":\"any\",\"messages\":[{\"role\":\"user\",\"content\":\"你好\"}]}"

# 方式 B：放到 URL 查询参数
curl "http://127.0.0.1:8199/url/gemini.com/v1/chat/completions?preset_name=pro" -d "..."

# 方式 C：放到请求体
{"model":"any","messages":[...],"stream":false,"preset_name":"pro"}
优先级规则

三处都传了预设时，按 路径里的预设 > URL 查询参数 > 请求体 的顺序生效。

查询参数与调度

参数示例作用
selector?selector=round_robinfirst_idle 优先空闲（默认）/ round_robin 轮询 / random 随机
tab_index?tab_index=2在域名路由下进一步锁定某个固定标签页
preset_name?preset_name=pro本次请求临时覆盖预设

完整调用示例

bash
# 1. 模型列表（连通性检查）
curl http://127.0.0.1:8199/v1/models

# 2. 默认聊天（自动分配标签页）
curl http://127.0.0.1:8199/v1/chat/completions ^
  -H "Content-Type: application/json" ^
  -d "{\"model\":\"any\",\"messages\":[{\"role\":\"user\",\"content\":\"你好\"}],\"stream\":false}"

# 3. 按域名路由 + 轮询调度
curl "http://127.0.0.1:8199/url/gemini.com/v1/chat/completions?selector=round_robin" -d "..."

# 4. 按域名路由 + 固定标签页
curl "http://127.0.0.1:8199/url/gemini.com/v1/chat/completions?tab_index=2" -d "..."

# 5. 域名 + 预设路径
curl http://127.0.0.1:8199/url/gemini.com/pro/v1/chat/completions -d "..."

# 6. 精确 URL 路由（token 从标签页池复制）
curl http://127.0.0.1:8199/tab-url/abc123def456/v1/chat/completions -d "..."

# 7. 精确 URL 路由 + 预设路径
curl http://127.0.0.1:8199/tab-url/abc123def456/pro/v1/chat/completions -d "..."

响应头诊断

路由类接口会返回诊断响应头，方便确认后端实际采用了哪种分配策略。

响应头含义
X-Tab-Selection-Mode实际选择模式：first_idle / round_robin / random / tab_index / exact_url
X-Requested-Route-Domain域名路由请求的目标域名；动态路由尚未选定标签页时也会返回
X-Resolved-Tab-Index已明确解析到固定标签页时返回的编号
X-Resolved-Route-Domain已解析标签页当前所属域名
X-Resolved-Exact-Url精确 URL 路由最终命中的完整 URL
X-Resolved-Preset-Name路径或参数最终命中的预设名称
动态域名路由说明

调用 /url/{domain}/... 且未传 tab_index 时，真正的标签页在工作流开始执行时才确定，因此这类响应只会带 X-Requested-Route-Domain 与 X-Tab-Selection-Mode，不会提前伪装成已解析到某个标签页。

登录状态与上下文上限

  • 推荐登录：登录你自己的账号，本地接口才能继承当前会话的站点能力与历史上下文。
  • 免登录亦可：若网站在未登录状态下允许对话，无需登录也能直接通过 API 调用。
上下文长度限制

受网站输入框单次可容纳字符上限约束，上下文不能太长，否则请求直接失败。长文本场景请开启「文件粘贴」。已测试单次最大发送长度：ChatGPT 约 200k 字符；Gemini 无会员约 30k，Pro 会员目前无明确上限；Arena AI 约 120k。

开发者协议兼容（实验性）

Claude Code 连通性支持

内置 Anthropic 风格协议的初步支持：兼容 /v1/messages、/v1/messages/count_tokens、/v1/models；除 Authorization 外亦解析 x-api-key，响应补充 request-id 诊断头。自 2026-05-28 起已完全适配。

Codex / Responses API（VSCode）

暴露实验性 /v1/responses 端点，自动转换并复用 /v1/chat/completions 调度逻辑（支持 input、instructions、tools 载荷）；流式模式生成 Codex 规范 SSE 事件，解决工具输出中断问题。

以上协议由于缺乏大规模工具用例覆盖，实际稳定性请按需测试。

提供系统级指标：全局健康度、当前已连接受控浏览器端口状态（默认 9222）以及站点配置、标签页池、实时日志、命令与全局设置的无缝切换。

动态管理

可以在此处实时为不同的站点、不同的标签页指派不同的预设，配置工作流拦截机制，并进行系统级参数覆盖调试。

请求异常调试与排查步骤

高并发或接口对接遇到瓶颈时，控制台是您的排障利器。请遵循以下标准诊断流程：

  1. 查看请求监控列表：点击失败记录，检查该请求是在哪个阶段失败的（如 acquire_tab_timeout 排队等待超时，还是 wait_first_response_timeout 首包响应超时）。
  2. 联合日志追踪：在日志页面中，通过请求的唯一 RequestID 或短追踪哈希，把该轮交互在 [ROUTE]（路由分发）、[POOL]（标签分配）、[INPUT]（粘贴输入）、[STREAM]（输出截获）以及 [NetworkMonitor] 模块的全部可爱化日志串联分析。
  3. 会话调试隔离：如果发现特定的标签页频繁出现卡死或慢速现象，可点击「标签页池」查看当前被锁定的页面状态，使用取消按钮强行中断或使用调试接口释放。
系统自愈与调试 API 接口
  • GET /api/system/request-history?limit=200 — 读取近期请求审计记录（含报错日志摘要）
  • GET /api/system/request-history/{request_id} — 读取单条请求详情（前端详情弹窗同款）
  • POST /api/debug/cancel-current — 取消当前正在处理的阻塞请求，支持指定 tab_id
  • POST /api/debug/force-release — 终极自愈接口。强行清空所有请求锁和标签页锁，重置池状态
06核心配置

选择器配置 Selectors

每个站点的每个预设都需要定义一组 CSS 选择器，告诉程序去哪里输入、点哪里发送、从哪里读回复。配置完全在控制台的可视化表单中回填，不需要修改任何 JSON 文件。这是所有配置里最关键、也最容易因网站改版而失效的部分。

零基础先认 3 个核心字段（在选择器配置面板顶部输入）

输入框选择器 (input_box)

指定聊天输入框（如 textarea）在页面中的位置。必填

发送按钮选择器 (send_btn)

网页上用于发送的按钮。必填

AI 回复容器 (result_container)

AI 回复文本渲染所在的外层大容器。必填 · 最易出错

控制台【选择器配置】表单提供了各字段的常用填写范例如下：

界面配置输入框常用填入示例（CSS选择器）定位技巧说明
输入框选择器 (input_box)textarea[id='prompt'] 或 textarea优先寻找含有 id 或专属 class 的 textarea 标签。
发送按钮选择器 (send_btn)button.send-btn 或 button[aria-label="发送消息"]选择含有发送小飞机图标或带有 Send 文本的 button 元素。
AI 回复容器 (result_container).markdown-body 或 div.message-content务必选择包裹整条 AI 回复文本的外层容器。如果选小了，会漏抓格式或发生字符截断。
新建对话按钮 (new_chat_btn)button.new-chat 或 a[href="/"]页面左上角或侧栏的新建对话按钮，按需补充。
怎么找得更稳
  • 在网页中右键点击目标元素（例如输入框） → 检查，查看其 HTML 属性。
  • 优先使用 id、data-testid、aria-label 这类具有明确语义的属性定位；如果 Class 名字是一长串无规则的随机字母数字（动态混淆 Class），先别用它，否则网站一刷新或改版就会失效。
  • 每一项输入完成后，点击右侧的【测试】按钮，确认能命中。

填写顺序建议

  1. 在受控浏览器中打开目标网站，右键输入框选择“检查元素”，把定位器填入 input_box。
  2. 再找到发送按钮，把其定位器填入 send_btn。
  3. 在对话框中发一条消息，等 AI 回复完毕后，找到包裹回复内容的最外层大 DIV，填入 result_container。
  4. 每填一项就点一次该字段右侧的「测试」，确认能命中——切忌全部填完再一起排查。
  5. 前三项稳定后，再考虑补充 new_chat_btn 或其他附件上传的选择器。

🛡️ 预设防风控与开头注入

在部分严格防爬的站点（如 GPT、Gemini、Arena 等）上，频繁发送相似模式的 Prompt 极易触发限频与风控。您可以通过控制面板开启防风控手段：

  • 开头追加随机占位片段：系统会在实际发送消息时，在用户 Prompt 前面自动追加一段无实际意义的前缀文本（如 `测试号，无实际意义: [随机混淆码]`）。您可以自定义“开头片段数”与“说明文本”。这可以让每条请求主体不一致，扰乱风控指纹。
  • 提示词内随机字符插入：系统会从您定义的候选字符集（例如：`abc123※☆`）中随机挑选一个字符，悄悄插入到用户提示词的随机位置，实现进一步的扰动防封。

🧠 输入框稳定与自愈等待

受控浏览器在点击新建对话后，部分网站的输入框 DOM 节点会重新创建，造成输入命令时因失焦或节点失效而粘贴失败。可以在站点控制台的【高级功能】中进行优化：

  • 输入框稳定等待：开启后，系统在执行 FILL_INPUT 操作前，会额外侦测输入框节点连续数次是否完全稳定。
  • 自愈选项：可以勾选“仅在刚点击 new_chat_btn 后启用”或“新建对话后等待 URL 切换”，并设置“最长等待秒数”（如 1.5 秒）与“URL 匹配正则”，确保页面就绪后再进行灌入。
  • 发送内容确认与自愈：开启后，系统在点击 send_btn 之后会做二次判定，确认输入框已被清空或字数明显缩短。若检测未通过（即提示词没发出去），系统会自动重试当前工作流，防止请求中断。
08核心配置

响应检测与自愈重试

系统需要判定 AI 什么时候开始输出（首次响应）、什么时候算真正说完，并从中捕获最新的流式文本。系统支持双模自动回退安全网和自动发送自愈重试机制。

检测模式流式输出容灾回退机制
DOM 模式 通用推荐✅监听网页 DOM 节点的字符变化，不需要了解站点接口，兼容性极高。
网络拦截模式✅（需适配）直接挂载 CDP 网络监听事件，阻断并获取 XHR/Fetch 数据包，还原代码和公式更精确。若首次响应超时，会自动向 DOM 模式回退。

🛡️ 网络监听超时自愈回退（双模安全网）

网络拦截模式虽然精准，但在网页改版、参数变化或解析器报错时可能会捕获失败，导致请求死锁白屏。系统配置中支持如下弹性策略：

  • 首次响应超时自动回退：网络监听在触发请求发送后，会开启 wait_first_response_timeout 超时侦测。如果在设定时间（默认 300s，在 stream_config 中的 network 项下）未检测到符合 listen_pattern 拦截规则的流量，系统会自动抛出超时异常，并无缝回退到 DOM 监控模式（StreamMonitor）继续捕获网页渲染出的文本，确保整个请求不因接口变动或匹配失效而挂起白屏。

⚡ 发送确认与自愈重试判定

在部分繁忙的 AI 平台上，点击发送按钮后偶发无响应。可以在站点控制台的【发送确认与重试】中定制自愈行为：

  • 自动再点一次（重试）：控制点击 send_btn 发送后，如果在设定的观察窗口内（如 1.8 秒）没有检测到输入框清空、明显的网络活动或文本生成状态，系统会启动重发，自动再次点击发送。
  • 重试频次：最大支持重试 2 次，两次重试间隔支持定制（如 0.6 秒）。
  • 安全防中断边界：一旦判定 AI 已经开始生成响应（或发送按钮已经变成了停止状态/正方形中断按钮），系统会自动强行禁止重试，防止二次点击将原本正在生成的长回复意外切断。

DOM 模式结束判定公式

稳定判定条件
生成结束判定 = (连续稳定次数 ≥ 稳定判定次数) AND (静默时间 > 静默超时阈值)

DOM 检查时，内容没有再变化，稳定计数增加，静默时间累加。两个阈值同时达到时，判定本轮回复完成。全局硬超时（如 300 秒）是两种模式共用的物理保底限时。

📦 独立 Cookie 标签页与多匿名窗口

对于像 arena.ai 这类需要严格隔离 Cookie（避免多标签页共用同一个登录会话或遭到风控判罚）的站点，系统在控制面板【高级功能】中提供了如下功能：

  • 自动接管手动新标签页：开启后，任何人工新建或被弹出的标签页均会被后台脚本接管，自动关闭原页面并转化为单独隔离的独立受控窗口。

🌐 域名路由分组 (Route Groups) — 企业级硬隔离

这是标签页池中最强大的多租户/多账号隔离机制。您可以为不同的请求定义专属的标签页路由组（在配置文件或标签面板管理）：

  • 工作逻辑：支持为指定域名（如 chat.deepseek.com）绑定一组专用的标签页，或者是根据传入的 URL token 指定独立会话。不同路由组的标签页互不干扰，从而实现多账号轮询分流、不同业务线硬隔离。
  • 分配策略 (allocation_mode)：支持 first_idle（最先空闲）、round_robin（同组轮询分摊）与 random（随机选择，防轨迹侦测）三种分发逻辑。
  • 错误标签页保留 (preserve_error_tabs)：当特定标签页内的交互发生了严重错误或风控拦截时，若勾选此项，系统会拒绝关闭并挂起该错误页面。控制台会亮起报警，方便管理员进入受控浏览器人工解密或排障。

控制台的【多模态提取】面板，统一负责管理如何从网页中提取 图片、音频、视频，并将其转换为标准 API 下游可以直接调用的本地 URL 或是 Markdown 图片标签。配置均可在前端页面直观完成，无需手动修改 JSON 配置文件。

💡 提示

如果您习惯了旧版本的代码级配置，系统会自动将您在控制台 UI 上的各项勾选与输入，无缝翻译并存入底层的 image_extraction 字段中。以下为您对照控制台前端 UI 各表单项的详细配置指南：

1. 基础开关与提取范围

  • 启用多模态提取（开关）：全局总开关。勾选后，系统才会在对话完成时去扫描网页里的图片/视频。
  • 提取策略模式（下拉框）：
    • 提取所有 (all) — 网页上生成的全部图片或媒体文件一并打包返回。
    • 仅首个 (first) — 仅提取网页回答中出现的第一张图片，适合标准的单图输出场景。
  • 媒体文件大小限制（数字输入框）：单位为 MB（默认 10）。超过此大小的媒体文件将被忽略，防止超大视频/无用底图占满服务器带宽。
  • 下载二进制资源 (Download Blobs)：开启后，自动下载网页里的二进制 Blob 媒体并转换为持久本地链接，防止因浏览器页面关闭导致临时链接失效。

3. 提取源定位（DOM 选择器配置）

控制台提供了【元素选择器】输入框，用来精确告诉脚本媒体在 HTML 中的哪个位置，以防误抓广告或侧边栏无用头像：

  • 媒体选择器 (Selector)：指示媒体标签。例如图片填 img，视频填 video, video source。
  • 容器选择器 (Container Selector)：非常重要。用来圈定提取范围。例如填 .result-content 或 .img-grid。系统将只抓取该容器范围内的媒体文件，完美避开页面顶栏、侧边栏或者无关图片。

4. 页面朗读音频捕获（TTS 语音录制）

除了抓取网页里已有的 audio 节点外，如果想录制 AI 的“在线朗读”发音，可以展开并配置【朗读捕获】表单：

  • 启用朗读录制 (audio_capture_enabled)：勾选后，系统会在文本输出完成后，去寻找并自动点击网页中的“朗读”或“发声”按钮来捕获音频。
  • 朗读按钮选择器 (audio_trigger_selector)：填入发音按钮的精确定位器。
  • 按钮文本兜底 (audio_trigger_labels)：当选择器不稳定时，可通过文字（如 "朗读", "发音", "收听"）匹配按钮。
  • 网络音频直抓配置 (url_patterns)：通过设置过滤关键词（如 tts、voice），若页面播放时有对应的网络音频数据流，系统会优先直抓网络流数据包并合成输出，抓取失败时自动降级为系统页面内录音兜底，保证音质无损。
效果产出与消费

提取成功后的图片会自动以 Markdown 标签格式 `![image](/download_images/xxx.png)` 返回；音频和视频会以播放器链接 `[audio_0](/download_images/xxx.mp3)` 的形式返回，各类 API 下游客户端（如 SillyTavern 等）均能一键直接消费和播放。

函数调用 Tool Calling

本反代兼容 OpenAI 标准的 tools 格式，能把系统外的函数定义，动态解释并以结构化的 XML 调用块（如 <adapter_calls> / <call> / <arg>）灌入网页端，再将 AI 输出解析回标准格式。

底层工作逻辑

这不是官方 API 原生的函数调用。后端会将工具 schema 翻译成高可读性的 Prompt 注入会话首尾，在捕获到 XML 片段后，解析参数并回调本地执行。若返回格式错误，后端会自动发起单步聚焦修复或完整上下文修复重试。

⚙️ 系统环境变量参数（设置 → 环境配置 → 函数调用）

高级用户可以直接通过以下环境变量或中控面板调优函数调用的稳定度和抗干扰行为：

  • 最大返回结果字符限制 (TOOL_CALLING_MAX_TOOL_RESULT_CHARS)：默认限制单次工具调用的返回内容为 300,000 字符。这能有效防止外部工具返回过于庞大的数据集时，直接将网页模型的 Token 窗口限制打满或引发崩溃。
  • 零宽字符噪声注入 (TOOL_CALLING_PROMPT_PADDING_ENABLED 与 TOOL_CALLING_PROMPT_PADDING_OBFUSCATE)：开启后，系统在注入工具提示词时，会随机混入零宽噪声字符（如 \u200b, \u200c, \u200d），打乱可能存在的自动化过滤检测特征，大幅提升模型在高风控站点上的执行隐蔽度。

两种内部修复策略

自愈策略适合场景工作特征

16上手指南

新增站点配置流程

当您准备接入一个新的未适配 AI 聊天网站时，系统提供了“AI 自动解析生成”与“纯手动点选配置”两条极为弹性的接入路线。

路线 A：辅助 AI 自动扫描分析 极速推荐

这是最轻松的接入方式，借助外部的 LLM 自动扫描并分析目标网页的 HTML，自动为您生成一套可用的初始选择器配置：

  1. 在受控浏览器里，新开标签页打开您想接入的 AI 聊天主页，并确保已经登录完成、停在对话输入页面。
  2. 对该站点发起第一次真实 API 请求（例如请求路由为：/url/example-ai.com/v1/chat/completions）。
  3. 后端系统检测到该域名是一个完全未配置的新站点，会自动捕获受控浏览器页面的 DOM 结构，并调用在设置中配置好的辅助 AI 接口进行页面分析。
  4. 分析通过后，后端会自动为您在该域名的站点下，生成初始的主预设，并把识别到的选择器与最短工作流自动写入底层。
重要细节

自动分析只有在对未知域名发起第一次真实 API 请求时才会触发。仅仅在控制面板上点击“新增站点”或者是只在浏览器里打开网页但不发 API 请求，辅助 AI 是不会启动扫描的。

如何确认识别成功
  • 在【日志】页面中，会亮起 [AI分析] 日志字样，最终显示“分析完成，配置已自动写入 sites.json”。
  • 刷新控制台，左侧站点导航列表中会多出这个新增的域名。
  • 进入该站点的配置面板，检查各选择器字段。务必仔细人工核对 result_container（AI回复容器），这是由于页面结构嵌套过深时，辅助 AI 最容易出现识别微调偏差的字段。

路线 B：手动在控制面板配置调试

如果您没有配置辅助 AI 密钥，或者目标页面带有强烈的 JS 混淆，可以直接通过控制台进行纯手工配置：

  1. 点击控制台左侧侧边栏下方的【新增站点】按钮，在弹出的窗口里输入目标站点的域名（如 chat.example.com）。
  2. 在左侧列表中点击新生成的域名，系统会自动为您初始化一个“空白主预设”。
  3. 去目标站点右键检查元素，将核心三项（input_box、send_btn、result_container）填入选择器表单中。
  4. 在【工作流】节点链中，可视化添加并拼装出您的最短可用交互步骤。
  5. 依次点击核心字段右侧的“测试”，调试无误后保存。

🖱️ 前端工作流配置面板拼装推荐步骤（最短可用工作流）

在工作流面板中，一个能够正常驱动绝大多数聊天网站的最简可用工作流节点顺序如下，您可以通过点击【＋ 增加步骤】进行拼接：

  1. CLICK 节点：目标填入 new_chat_btn，勾选“可选（Optional）”，以防首次打开本就处于新对话时点击失败。
  2. WAIT 节点：时值设为 0.5 秒，给页面重置预留少许过渡时间。
  3. FILL_INPUT 节点：目标填入 input_box。这是将用户提示词灌入输入框的必需步骤。
  4. CLICK 节点：目标填入 send_btn，用于点击发送按钮。
  5. KEY_PRESS 节点：目标填入 Enter，勾选“可选”，可在部分回车也能发送的网站上作为备用指令。
  6. STREAM_WAIT 节点：目标填入 result_container。用于捕获流式输出。必须以此步骤作为整条工作流的最终结尾。

手动调试推荐顺序：建议您按 `输入框` → `发送按钮` → `回复容器` 的顺序单步调试选择器。三项都亮绿灯后，再开始运行工作流。跑通一轮基本对话后，最后再来调精细的文件粘贴阈值与多模态提取参数，这种梯度调试效率最高。

两个必须知道的坑
  1. Chrome 136+ 安全调整：默认 Chrome 数据目录不再接受 --remote-debugging-port。把 BROWSER_PROFILE_DIR 直接写成系统默认 User Data，浏览器会被打开但调试端口起不来，脚本无法接管。见 Chrome 官方说明。
  2. 会话数据风险：浏览器用户目录可能包含登录态、Cookie、本地缓存和扩展数据。只建议复制你自己的本机目录副本用于本机调试；不要打包分享、提交仓库或发给别人。
  1. 先完全关闭所有 Chrome 窗口。
  2. 新建专用目录，如 C:\Users\你的用户名\AppData\Local\UniversalWebApiProfile。
  3. 从原始 ...\Google\Chrome\User Data 中至少复制 Local State 和要使用的配置子目录（如 Default、Profile 1）到新目录。
  4. 在 .env 指向这份副本，重新运行 start.bat。

容易写错的地方

BROWSER_PROFILE_DIR 不要写成 ...\User Data\Default——它应该指向 User Data 这一层的"根目录"；具体用哪个子配置文件夹，写在 BROWSER_PROFILE_NAME（常见值 Default、Profile 1、Profile 2）。

不在乎沿用登录态？

最省心：留空 BROWSER_PROFILE_DIR，让项目继续使用根目录的 chrome_profile/。隔离最彻底，也最不容易和日常 Chrome 互相影响——对大多数用户这是默认推荐方案。

配置管理与更新白名单

  • 保存：控制面板修改后点「保存」立即写入配置文件；大部分配置热重载即刻生效（环境配置除外）。
  • 导入 / 导出：支持导出全量配置或单个站点配置为 JSON，也可导入。预设随站点配置一并导出。
  • 备份：建议定期备份 config/ 目录。

更新白名单（控制面板 → 设置 → 更新白名单，保存后写入 config/update_settings.json，下次自动更新时生效）控制的是"更新时是否保留不变"，不是运行时开关：

  • 勾选 config/、app/、static/ 这类目录项时保留整个目录。
  • config/sites.json 与 config/commands.json 未勾选时默认按更新器规则合并（保留本地站点、命令与状态，补入新版内容）；勾选则整份保持本地版本不动。
  • config/update_settings.json 会被自动保留，无需手动勾选。
当前默认保留项

config/sites.local.json（站点默认预设覆盖）、config/commands.local.json（命令启用状态与分组覆盖）、chrome_profile/（登录态，仅本机自用）、venv/、logs/、image/、updater.py、.git/、__pycache__/、*.pyc、backup_*/。只保留登录态和个人配置时保持默认勾选即可；要保留自己改过的代码时再勾对应目录。

常见问题

酒馆显示「连接失败」怎么办？

酒馆的「测试 API」按钮发送的请求通常太短，部分站点（尤其 AI Studio）不会正常响应而误报失败。不要点测试按钮，直接发送一句真实对话来测试。

请求频繁失败，怎么排查？
  1. 脚本运行时操作了受控浏览器：工作流执行中不要点击、切换或操作受控浏览器，最多只能看着。
  2. 缩小窗口或折叠页面元素：可以最小化，但不要缩小窗口边框或折叠脚本要操作的区域。
  3. 保留了无关标签页：只留目标 AI 网站；出现异常重启脚本即可。
  4. 网站状态异常：看受控浏览器是否触发人机验证、被内容风控拦截，或单次输入太长被拒。
  5. 网络波动 / 代理切换 IP：表现为后台显示发送成功但浏览器没真正发出。可等约 5 分钟或直接重启脚本。
  6. 站点配置过期：前面都排除后，可能是站点改版导致选择器失效——提 Issue、加群反馈，或手动更新选择器。
同时打开多个标签页，脚本怎么处理？
  • 默认路由 /v1/chat/completions：自动挑空闲标签页。
  • 指定站点 /url/gemini.com/v1/chat/completions：只发到对应站点。
  • 指定标签页 /tab/1/v1/chat/completions：精确使用编号 1。
  • 可以给不同标签页配不同预设（一个聊天、一个识图）；多个请求到来时空闲标签页依次分配，全忙时排队。
标签页池里看不到新打开的标签页？
  • 确认页面已加载完成；chrome://newtab 这类空白页不会被识别。
  • 等 2–3 秒后点击控制台的「立即刷新」。
  • 还是不出现，直接重启脚本。
控制面板里的参数需要手动调整吗？

大多数时候不需要。默认值已覆盖最常见场景，遇到具体问题时再针对性调整即可。

目前有哪些已知问题？
  • VSCode 的 Codex 插件及 Claude Code 均已完全适配（自 2026-05-28 起），连接方法见「连接 API」。
  • DrissionPage 每次升级后都需要重新执行对应补丁。
  • "独立 Cookie 标签页"仍是高级实验功能——强行追求"同一受控窗口内的独立 Cookie" 在浏览器原生菜单等场景可能不稳，当前建议优先使用共享 Cookie 方案。
代理切换命令不生效怎么办？
  1. 确认 Clash 正在运行且启用了 External Controller。
  2. 确认 API 地址正确（默认 http://127.0.0.1:9090，有些客户端用别的端口）。
  3. 确认代理组名称拼写：访问 http://127.0.0.1:9090/proxies 查看实际名称。
  4. Clash 配了 secret 的话，命令配置里也要一起填。
  5. 在控制台「日志」搜索 [CMD] 开头的条目，看具体报错或切换结果。
复用自己的 Chrome 登录态，为什么配了不生效？
  • 直接指向了系统默认 User Data：Chrome 136+ 上远程调试端口会起不来——先复制一份独立目录，再让脚本使用副本。
  • BROWSER_PROFILE_DIR 层级写错：应指向 User Data 这一层，而不是 ...\User Data\Default；子配置文件夹写在 BROWSER_PROFILE_NAME。
  • 改完没重启：浏览器启动相关配置只有重新运行 start.bat 才生效。

具体步骤回看「系统设置 → 浏览器登录态复用」。
